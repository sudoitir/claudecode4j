# Contributing to ClaudeCode4J

First off, thank you for considering contributing to **ClaudeCode4J**! It's people like you that make open source
projects such amazing places to learn, inspire, and create.

This guide will help you get started with the contribution process.

## üêõ Bug Reports

If you spot a problem, please search the [Issues](https://github.com/sudoit/claudecode4j/issues) first to ensure it
hasn't already been reported. If it's a new issue, please open a new bug report and include:

1. **Expected Behavior:** What did you expect to happen?
2. **Actual Behavior:** What actually happened?
3. **Reproduction Steps:** Detailed steps to reproduce the issue.
4. **Environment:** Java version, OS, library version, etc.

## üí° Feature Requests

If you have an idea for a new feature, please open an Issue to discuss it before you start coding. This ensures that
your work aligns with the project's roadmap and prevents wasted effort.

## üõ† Pull Request Process

We follow the standard GitHub Flow:

1. **Fork:** Fork the repository to your own GitHub account.
2. **Clone:** Clone the project to your local machine.
3. **Branch:** Create a new branch for your specific change.

   ```bash
   git checkout -b feature/my-new-feature
   # or for bugs
   git checkout -b fix/bug-description
   ```
4. **Code:** Implement your changes.
5. **Test:** Ensure all tests pass.

   ```bash
   mvn test
   ```
6. **Commit:** Commit your changes with a clear message. (We
   recommend [Conventional Commits](https://www.conventionalcommits.org/)).
7. **Push:** Push the changes to your fork.
8. **Pull Request:** Submit a Pull Request to the `master` branch of the original repository.

## üìù Coding Standards

* Follow standard Java coding conventions.
* Add **Javadoc** for new classes and public methods.
* If you add a new feature, please add a corresponding **Unit Test**.
* Keep the code clean and readable.
